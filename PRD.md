# PRD — Система карго-консолидации

> Product Requirements Document v1.1
> Дата: 2026-03-25
> Платформа: Web + Telegram Bot (без мобильного приложения)

---

## 1. Обзор продукта

### 1.1 Что это
Веб-платформа для карго-компании, которая принимает посылки клиентов с китайских маркетплейсов (Taobao, 1688, Pinduoduo, Poizon и др.) на свои склады в Китае, консолидирует их в коробки и отправляет в Таджикистан и страны СНГ.

### 1.2 Проблема
Клиенты покупают товары на китайских маркетплейсах, но прямая доставка в Таджикистан/СНГ дорогая, медленная или недоступна. Карго-компания решает эту проблему — собирает посылки на складе, объединяет и отправляет оптовым грузом.

Сейчас процесс ведётся вручную (Excel, WhatsApp, бумажные записи), что приводит к:
- потере посылок и путанице с клиентами
- невозможности отследить статус
- ошибкам в расчёте стоимости
- отсутствию прозрачности для клиентов

### 1.3 Решение
Единая система с веб-кабинетом (для клиентов, работников, админов) и Telegram-ботом, которая автоматизирует весь цикл: от регистрации посылки на складе до выдачи клиенту в Душанбе.

### 1.4 Каналы доступа
| Канал | Для кого | Назначение |
|-------|---------|-----------|
| Веб (адаптивный, mobile-first) | Все роли | Полный функционал |
| Telegram-бот | Клиенты | Трекинг, QR, уведомления, быстрые действия |

### 1.5 Ключевые метрики успеха
- Время приёмки посылки на складе: < 2 мин (скан QR + фото + вес)
- Процент потерянных/перепутанных посылок: → 0%
- Клиент может отследить статус 24/7 без звонка оператору
- Среднее время от "Собрать коробку" до отправки: < 48 часов

## 2. Пользователи и роли

### 2.1 Клиент (customer)
Физическое лицо или мелкий предприниматель, покупающий товары на китайских маркетплейсах.

**Потребности:**
- Получить уникальный ID и адрес склада для указания при покупке
- Видеть какие посылки пришли на склад
- Решить когда отправлять (кнопка "Собрать коробку")
- Отслеживать коробку на всём маршруте
- Знать стоимость до отправки
- Получать уведомления о каждом этапе
- Оплатить при получении

### 2.2 Работник склада (warehouse_worker)
Сотрудник на складе в Гуанчжоу, Урумчи или Душанбе. **Привязан к конкретному складу** — видит только свои данные.

**Потребности:**
- Быстро принять посылку (скан → фото → вес → полка)
- Идентифицировать к какому клиенту относится посылка
- Работать с неопознанными посылками
- Фиксировать повреждения при приёмке
- Отклонять запрещённые товары
- Собрать и упаковать коробку по запросу клиента
- Печатать этикетки для коробок
- Оформить рейс (погрузка коробок)
- Выдать коробку клиенту и принять оплату (Душанбе)

### 2.3 Администратор (admin)
Руководство компании.

**Потребности:**
- Дашборд реального времени (сегодня: принято, отправлено, выручка)
- Видеть все заказы, посылки, коробки по всем складам
- Глобальный поиск по любому коду
- Управлять тарифами и курсами валют
- Управлять сотрудниками и складами
- Финансовые отчёты с экспортом в Excel
- Статистика по маршрутам и загрузке
- Настройки компании (адреса складов, лого, правила хранения)

### 2.4 B2B-клиент (organization)
Мелкий опт — предприниматель с регулярными заказами.

**Отличия от обычного клиента:**
- Специальные тарифы / скидки
- Возможность кредитного лимита (оплата позже)
- Общий кабинет на организацию

## 3. Бизнес-процесс (детально)

### 3.1 Регистрация клиента
1. Клиент регистрируется (телефон + SMS-код ИЛИ через Telegram-бот)
2. Система генерирует уникальный `client_code` (формат: `CD-XXXX`)
3. Клиент получает:
   - QR-код (для вложения в посылку или записи на ней)
   - Адреса складов (Гуанчжоу и/или Урумчи)
   - Инструкцию: "При покупке укажите этот адрес и ваш код CD-XXXX"
4. Клиент выбирает "домашний склад" (основной склад для отправок)

### 3.2 Покупка и доставка на склад
1. Клиент покупает товар на маркетплейсе (Taobao, 1688, Pinduoduo, Poizon и др.)
2. Указывает адрес склада карго-компании + свой `client_code`
3. (Опционально) Клиент добавляет трек-номер маркетплейса в систему для ожидания
4. Маркетплейс доставляет посылку на склад

### 3.3 Приёмка посылки на складе
1. Работник получает посылку
2. Ищет `client_code` на посылке (визуально или сканирует QR)
3. **Если код найден** → В системе создаёт запись:
   - Сканирует/вводит client_code → привязка к клиенту
   - Выбирает маркетплейс-источник (Taobao / 1688 / Pinduoduo / Poizon / Другое)
   - Фотографирует посылку (минимум 1 фото)
   - Взвешивает (вес в кг)
   - Замеряет габариты (Д × Ш × В в см) → система считает объёмный вес
   - Выбирает категорию товара (одежда, электроника, обувь, косметика, прочее)
   - Добавляет описание ("кроссовки Nike Air Max")
   - Отмечает "хрупкое" если нужно
   - **Если повреждение** → отмечает "повреждено" + фото повреждения + комментарий
4. **Если код НЕ найден** → Создаёт как "неопознанную посылку" (см. 3.3.1)
5. **Если запрещённый товар** → Отклоняет (см. 3.3.2)
6. Система присваивает ячейку хранения (напр. A-03-12)
7. Клиент получает Telegram-уведомление: "Ваша посылка принята на складе Гуанчжоу"

#### 3.3.1 Неопознанные посылки
1. Работник создаёт запись без привязки к клиенту
2. Заполняет: фото, вес, размеры, описание, телефон получателя (если есть на посылке)
3. Посылка попадает в список "Неопознанные" в панели работника и админа
4. Система пытается найти клиента по телефону получателя (если указан)
5. Работник/админ может вручную привязать к клиенту
6. Клиент в личном кабинете видит раздел "Ожидают привязки" и может заявить посылку
7. Если посылка не востребована N дней — уведомление админу

#### 3.3.2 Запрещённые товары
- Список запрещённых товаров настраивается админом (батарейки, жидкости, оружие и т.д.)
- Работник отмечает посылку как "отклонена" + причина
- Клиент получает уведомление с причиной отказа
- Посылка не принимается на хранение

### 3.4 Хранение
- Посылка хранится на складе до решения клиента
- Бесплатный период: N дней (настраивается в тарифе, по умолчанию 30)
- После — платное хранение (за день)
- Система отправляет напоминания:
  - За 5 дней до конца бесплатного периода
  - В день начала платного хранения
  - Каждые 15 дней после (если не забрали)
- **Невостребованные посылки**: если посылка хранится > 90 дней — уведомление админу, клиенту предупреждение об утилизации

### 3.5 Консолидация ("Собрать коробку")
1. Клиент заходит в кабинет → видит свои посылки на складе
2. **Важно:** посылки группируются по складу — объединить можно только посылки с ОДНОГО склада
3. Выбирает какие посылки объединить → (опционально) пишет примечание ("упакуйте аккуратно", "хрупкое сверху")
4. Нажимает "Собрать коробку"
5. Система проверяет:
   - Все посылки на одном складе
   - Суммарный вес не превышает лимит (настраиваемый, по умолчанию 50 кг)
   - Нет посылок со статусом "повреждено" без подтверждения клиента
6. Система создаёт `consolidation_box` со статусом `requested`
7. Система предварительно считает стоимость (сумма весов × тариф)
8. Работник склада видит запрос в очереди на упаковку
9. Работник:
   - Собирает посылки из ячеек
   - Упаковывает в коробку
   - Взвешивает коробку (итоговый вес)
   - Замеряет габариты коробки → объёмный вес
   - Система считает: `billable_weight = MAX(фактический, объёмный)`
   - Система считает: `price = MAX(billable_weight × rate, min_price)`
   - Работник подтверждает → статус `packed`
10. Клиент получает уведомление: "Коробка собрана. Вес: 12.5 кг. Стоимость: $XX. Ожидайте отправку."

#### 3.5.1 Отмена запроса на консолидацию
- Клиент может отменить запрос, пока статус `requested` (работник ещё не начал)
- Если статус `packing` — отмена невозможна, нужно обратиться к оператору
- При отмене: посылки возвращаются в статус `stored`

### 3.6 Формирование рейса
1. Когда набирается достаточно коробок — работник/админ создаёт рейс (shipment_batch)
2. Выбирает коробки для загрузки → привязывает к рейсу
3. Указывает маршрут: Гуанчжоу → Урумчи → Душанбе (или Гуанчжоу → Душанбе)
4. Указывает транспорт (номер машины / контейнера)
5. Система генерирует **манифест рейса** (PDF) — список всех коробок с весами
6. Отправляет рейс → все коробки получают статус `in_transit`
7. Все клиенты получают уведомление: "Ваша коробка отправлена"

### 3.7 Транзит и трекинг
- Работник обновляет статус рейса по маршруту:
  - `in_transit` — в пути
  - `customs` — на таможне
  - `arrived` — прибыл на склад назначения
- При каждом обновлении — уведомление всем клиентам коробок в рейсе
- Клиент видит текущее положение в личном кабинете (таймлайн)

### 3.8 Прибытие и выдача (Душанбе)
1. Рейс прибыл → работник Душанбе отмечает `arrived`
2. Коробки распределяются → каждая получает `shelf_location`
3. Статус коробки: `ready` (готова к выдаче)
4. Клиент получает уведомление: "Ваша коробка готова к получению. Адрес: ... Стоимость: $XX"
5. Клиент приходит на склад
6. Работник находит коробку по коду / QR клиента
7. Клиент оплачивает (наличные / перевод / карта)
8. Система пересчитывает стоимость в валюте оплаты по текущему курсу
9. Работник отмечает оплату и выдачу → статус `delivered`
10. Система генерирует квитанцию (PDF)
11. Клиент получает уведомление: "Спасибо! Коробка выдана."

## 4. Функциональные требования

### F1. Аутентификация и профиль
- **F1.1** Регистрация по номеру телефона + SMS OTP
- **F1.2** Вход по номеру телефона + SMS OTP (без пароля)
- **F1.3** Привязка Telegram-аккаунта (по deep link из бота)
- **F1.4** Генерация уникального `client_code` при регистрации
- **F1.5** Генерация QR-кода на основе `client_code`
- **F1.6** Профиль: ФИО, телефон, email, язык (рус/тадж)
- **F1.7** Выбор "домашнего склада" (откуда чаще отправляет)
- **F1.8** Роли: customer, warehouse_worker, admin
- **F1.9** Работник привязан к конкретному складу (видит только свой)

### F2. Личный кабинет клиента (Web)
- **F2.1** Дашборд: кол-во посылок на складе (по каждому складу), коробки в пути, готовые к выдаче
- **F2.2** Список посылок с фильтрами (статус, склад, дата, категория)
- **F2.3** Детали посылки: фото, вес, размеры, описание, категория, маркетплейс, статус, дата приёмки
- **F2.4** Добавление трек-номера маркетплейса вручную (для ожидания посылки)
- **F2.5** "Мой QR-код" — экран с QR и client_code для показа/печати/скачивания PDF
- **F2.6** "Адреса складов" — адреса с инструкцией, кнопка "Копировать адрес"
- **F2.7** Список коробок с фильтрами (статус, дата)
- **F2.8** Детали коробки: список посылок внутри, вес (факт/объёмный/оплачиваемый), стоимость, трекинг-таймлайн
- **F2.9** Кнопка "Собрать коробку" — выбор посылок для консолидации (только с одного склада)
- **F2.10** Предварительный расчёт стоимости перед подтверждением
- **F2.11** Поле примечания при сборке коробки ("упакуйте аккуратно")
- **F2.12** Отмена запроса на сборку (пока статус = requested)
- **F2.13** Валидация: нельзя объединить посылки с разных складов; предупреждение о повреждённых посылках
- **F2.14** "Ожидают привязки" — список неопознанных посылок, которые клиент может заявить
- **F2.15** История оплат
- **F2.16** Калькулятор стоимости (публичный, без авторизации тоже)

### F3. Панель работника склада (Web, mobile-first)
- **F3.1** Приёмка посылки: скан QR / ввод client_code → маркетплейс → фото → вес → размеры → категория → описание
- **F3.2** Отметка повреждения при приёмке: чекбокс "повреждено" + фото повреждения + комментарий
- **F3.3** Отклонение запрещённого товара: причина → уведомление клиенту
- **F3.4** Приёмка неопознанной посылки (без client_code): фото + вес + телефон с посылки
- **F3.5** Список неопознанных посылок + ручная привязка к клиенту
- **F3.6** Поиск клиента по client_code, телефону, имени
- **F3.7** Список посылок на складе (фильтр по статусу, клиенту, дате, категории)
- **F3.8** Назначение ячейки хранения
- **F3.9** Очередь на консолидацию (запросы "Собрать коробку" с примечаниями клиентов)
- **F3.10** Процесс упаковки: собрать посылки → упаковать → взвесить → замерить → подтвердить
- **F3.11** Печать этикетки коробки (код, клиент, маршрут, вес)
- **F3.12** Формирование рейса: выбрать коробки → создать batch → указать транспорт
- **F3.13** Генерация и печать манифеста рейса (PDF: список коробок + веса)
- **F3.14** Обновление статуса рейса (в пути / таможня / прибыл)
- **F3.15** Выдача коробки: найти по коду → показать стоимость (в валюте оплаты) → принять оплату → отметить выдачу
- **F3.16** Генерация квитанции оплаты (PDF)
- **F3.17** Работник видит ТОЛЬКО данные своего склада

### F4. Тарификация и валюты
- **F4.1** Расчёт по фактическому весу (кг)
- **F4.2** Расчёт объёмного веса: `(Д × Ш × В) / делитель` (делитель настраиваемый, по умолчанию 6000)
- **F4.3** Оплачиваемый вес: `MAX(фактический, объёмный)`
- **F4.4** Стоимость: `MAX(оплачиваемый_вес × цена_за_кг, мин_цена)`
- **F4.5** Тарифы по маршрутам (Гуанчжоу→Душанбе, Урумчи→Душанбе)
- **F4.6** Валюты: USD, CNY, TJS, RUB
- **F4.7** Таблица курсов валют (обновляется админом вручную)
- **F4.8** Автоконвертация: тариф в USD → показ клиенту в его валюте
- **F4.9** B2B-тарифы: скидка % или спец-цена за кг для организаций
- **F4.10** Платное хранение: бесплатный период (дни) + цена за день
- **F4.11** Доп. услуги: хрупкое (+%), страховка, упаковка
- **F4.12** Лимит веса коробки (настраиваемый, по умолчанию 50 кг)

### F5. Уведомления (Telegram)
- **F5.1** Посылка принята на складе (+ фото)
- **F5.2** Посылка повреждена при приёмке (+ фото повреждения)
- **F5.3** Посылка отклонена (запрещённый товар) + причина
- **F5.4** Коробка собрана (+ вес + стоимость)
- **F5.5** Коробка отправлена (+ маршрут + ожидаемые сроки)
- **F5.6** Коробка на таможне
- **F5.7** Коробка прибыла в Душанбе
- **F5.8** Коробка готова к выдаче (+ адрес склада + стоимость)
- **F5.9** Напоминание о хранении (приближается платный период)
- **F5.10** Предупреждение о невостребованной посылке (> 90 дней)
- **F5.11** Оплата принята / коробка выдана
- **F5.12** Неопознанная посылка — возможно ваша (если нашли совпадение по телефону)
- **F5.13** Язык уведомлений: русский или таджикский (по настройке)

### F6. Telegram-бот
- **F6.1** `/start` — регистрация или привязка аккаунта
- **F6.2** Главное меню (inline-кнопки): Посылки | Коробки | QR-код | Адреса | Калькулятор
- **F6.3** Мои посылки (список с кнопками "подробнее")
- **F6.4** Мои коробки (список с кнопками "отследить")
- **F6.5** `/track <код>` — отследить коробку по коду
- **F6.6** QR-код — показать как изображение
- **F6.7** Адреса складов (с кнопкой "Копировать")
- **F6.8** Калькулятор стоимости (диалог: маршрут → вес → результат)
- **F6.9** Смена языка (рус/тадж)
- **F6.10** Помощь / FAQ

### F7. Админ-панель (Web)

#### F7.A Дашборд реального времени
- **F7.1** Виджеты: сегодня принято посылок | собрано коробок | отправлено рейсов | выручка за сегодня
- **F7.2** Виджеты: всего посылок на складах (по каждому) | коробок в пути | ожидают выдачи
- **F7.3** Графики: приёмка за неделю/месяц, выручка за неделю/месяц

## 5. Нефункциональные требования

### NF1. Производительность
- Время отклика API: < 500 мс (95-й перцентиль)
- Загрузка фото: < 3 сек на фото (сжатие на клиенте)
- Поддержка: до 10 000 активных клиентов

### NF2. Безопасность
- Аутентификация: JWT + refresh tokens
- SMS OTP: срок действия 5 мин, макс. 5 попыток
- Разграничение по ролям (RBAC): customer, worker, admin
- Работник видит только данные своего склада
- Фото хранятся в приватном S3-бакете (pre-signed URLs)
- HTTPS обязателен
- Логирование действий сотрудников (audit log)
- Rate limiting на auth-эндпоинтах

### NF3. Локализация (i18n)
- Русский язык (основной)
- Таджикский язык
- Переключение языка в профиле и Telegram-боте
- Уведомления на выбранном языке
- UI: все тексты через файлы перевода (не хардкод)

### NF4. Доступность
- Веб-кабинет: адаптивный (mobile-first)
- Панель работника: оптимизирована для телефона (основной инструмент на складе)
- Работа при медленном интернете (оптимизация изображений, lazy load)
- Telegram-бот как основной канал для клиентов (низкий порог входа)

### NF5. Надёжность
- Бэкапы БД: ежедневно
- Мониторинг: uptime, ошибки API, очередь уведомлений
- Graceful degradation: если Telegram API недоступен — уведомления в очередь

## 11. Риски и допущения

### Риски
| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| Плохой интернет на складе в Китае | Высокая | Среднее | Сжатие фото на клиенте, очередь загрузки, оптимистичный UI |
| Клиенты не пишут client_code на посылке | Высокая | Высокое | Инструкция при регистрации, workflow неопознанных посылок, поиск по телефону |
| Повреждение посылки при транспортировке | Средняя | Среднее | Фото при приёмке (доказательство), отметка повреждения, страховка (будущее) |
| Задержки на таможне | Средняя | Среднее | Статус "customs" + уведомление клиента |
| Потеря посылки между складами | Низкая | Высокое | Сканирование при каждой перегрузке, манифест рейса, аудит-лог |
| Курс валют меняется между расчётом и оплатой | Средняя | Низкое | Тариф в USD, конвертация только при выдаче по актуальному курсу |

### Допущения
- SMS-провайдер для Таджикистана доступен (для OTP)
- У работников склада есть смартфон с камерой и браузером
- Клиенты используют Telegram (основной мессенджер в Таджикистане)
- Начальный объём: до 1000 посылок/месяц (MVP не требует горизонтального масштабирования)
- Курсы валют обновляются админом вручную (не автоматически)
